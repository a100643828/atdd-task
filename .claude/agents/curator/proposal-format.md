## 來源標註規範

### 必填欄位

每個提案項目**必須**包含：

| 欄位 | 說明 | 必填 |
|------|------|------|
| `來源:` | 知識的出處，使用來源標籤 | 是 |
| `推理鏈:` | 僅 `[推導]` 項目需要 | `[推導]` 時必填 |
| `不確定:` | 信心度 < 95% 時說明原因 | 信心度 < 95% 時必填 |

### 來源標籤格式

```
[文件] {path} § {section}          — 引用知識文件的特定章節
[code] {path}:L{line}              — 引用程式碼的特定行
[用戶] Q{N}                        — 引用訪談第 N 輪的回答
[推導] A + B → C                   — 從多個來源推導
[待確認]                            — Phase 4 必須解決
```

### 複合來源

一個項目可以有多個來源，用 `+` 連接：

```
來源: [用戶] Q2 + [code] app/models/foo.rb:L25
來源: [文件] strategic/Bar.md § 商務依賴 + [用戶] Q3
```

## 變更類型說明

| 類型 | 說明 |
|------|------|
| `[新增]` | 全新的術語/規則/章節 |
| `[修正]` | 修改現有內容 |
| `[刪除]` | 移除過時內容 |
| `[更新]` | 章節內容更新 |

## 信心度標記

| 信心度 | 標記 | 處理方式 |
|--------|------|----------|
| `>= 95%` | ✅ | 可直接寫入 |
| `80-94%` | ⚠️ | 需要用戶確認 |
| `< 80%` | ❌ | 必須詢問後才能寫入 |

## 跨域提案額外內容

處理兩個 Domain 時，提案可能包含：

1. **domain-map.md 的 Context Mapping 更新**
2. **兩個 strategic/{domain}.md 的商務依賴更新**
3. **兩個 tactical/{domain}.md 的 Integration 技術細節更新**
4. **建議的 ACL（Anti-Corruption Layer）設計**
5. **建議的 Domain Event 定義**
